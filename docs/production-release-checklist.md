# قائمة إطلاق المحرر إلى الإنتاج

هذه القائمة مطلوبة بعد دمج أي تحديث يؤثر في المحرر أو الحفظ أو النشر.

## 1. النشر

- تأكد أن Render نشر أحدث commit من `main`.
- افتح `/api/health` وتحقق من `status`, `version`, `release`.
- افتح `/readyz` وتأكد أن الحالة `ok` وأن فحصي قاعدة البيانات والتخزين `ready`.
- زامن ملفات الواجهة إلى `public_html/nfc/` عند استخدام cPanel/LiteSpeed.
- عند إطلاق مدير الأصول تأكد من مزامنة `editor-asset-manager.js` و`editor-asset-manager.css` مع `runtime-config.js` و`sw.js`.
- نفذ تحديثًا عاديًا وتأكد أن Service Worker يستخدم إصدار الكاش الموجود حاليًا في `sw.js`.
- تأكد أن `runtime-config.js` يعرض رقم الإصدار الموجود في المستودع، وأن ملفات الشريط ومدير الأصول تُحمّل من أول زيارة.

## 2. الفحص الآلي للنسخة الحية

بعد اكتمال نشر Render ومزامنة cPanel شغّل:

```bash
npm run verify:production
```

يفحص الأمر تلقائيًا:

- المحرر العربي والإنجليزي.
- تطابق رقم إصدار `runtime-config.js` مع النسخة الموجودة في `main`.
- وصول ملف إصلاح الشريط العلوي وتطبيق علامات التنسيق المطلوبة.
- وصول ملفات مدير الأصول الجديدة وعلامات المعالجة والسحب والقص.
- تطابق Service Worker وإصدار الكاش.
- `/healthz` و`/readyz` و`/api/health` على Render.
- جاهزية قاعدة البيانات ومساحة رفع الملفات.

يُحفظ التقرير في `production-verification.json`. ويمكن تشغيل نفس الفحص من GitHub Actions عبر workflow باسم **Production smoke verification**. يعمل الفحص أيضًا يوميًا لاكتشاف رجوع الموقع إلى نسخة قديمة أو تعطل البنية التحتية.

يمكن تغيير النطاقات عند الحاجة:

```bash
PUBLIC_ORIGIN=https://www.mcprim.com \
API_ORIGIN=https://nfc-vjy6.onrender.com \
npm run verify:production
```

## 3. رحلة المستخدم الأساسية

نفذها بالعربية والإنجليزية:

1. تسجيل الدخول بالبريد.
2. تسجيل الدخول عبر Google.
3. إنشاء تصميم جديد.
4. تعديل الاسم والوظيفة ووسائل الاتصال.
5. رفع شعار وصورة عبر اختيار الملف وعبر السحب والإفلات.
6. التأكد من ظهور حالة المعالجة، وتحذير الدقة المنخفضة عند استخدام صورة صغيرة.
7. تجربة التدوير والتكبير وإعادة الضبط داخل نافذة القص.
8. تعديل ترتيب الطبقات والموضع والحجم والدوران والشفافية.
9. الحفظ وإغلاق الصفحة.
10. إعادة فتح التصميم والتأكد من بقاء جميع التعديلات.
11. المعاينة ثم النشر.
12. فتح رابط Viewer ومقارنته بالمحرر.
13. اختبار PNG وPDF وVCF وQR.

## 4. اختبارات فقدان البيانات

- عدّل حقلًا وتأكد من ظهور «تغييرات غير محفوظة».
- حاول إغلاق التبويب وتأكد من ظهور تحذير المتصفح.
- اقطع الإنترنت، عدّل التصميم، وتأكد من «محفوظ محليًا».
- أعد تحميل الصفحة وتأكد من ظهور خيار استعادة النسخة المحلية.
- أعد الاتصال وتأكد من إعادة محاولة الحفظ السحابي.
- نفذ تعديلات متتابعة سريعة وتأكد أن آخر تعديل هو المحفوظ.
- اختبر فشل API وتأكد أن النسخة المنشورة السابقة لا تتأثر.

## 5. المتصفحات والأجهزة

- Chrome Desktop.
- Chrome Android.
- Safari Desktop/WebKit.
- Safari iPhone.
- شاشة صغيرة ومتوسطة وكبيرة.
- الوضع الفاتح والداكن.
- السحب والإفلات على سطح المكتب واختيار الصور من الهاتف.

## 6. شروط الموافقة

لا يُعتمد الإصدار إلا إذا:

- نجحت اختبارات Jest وESLint وnpm audit.
- نجحت Playwright في Chromium وWebKit للهاتف وسطح المكتب.
- نجح `npm run verify:production` بعد نشر الواجهة والخادم.
- نجحت معالجة الصور والقص والرفع دون تشغيل معالج الرفع القديم مرتين.
- تطابق Viewer مع المحرر.
- لم تظهر أخطاء JavaScript حرجة.
- نجحت الاستعادة المحلية والحفظ بعد عودة الشبكة.
- عاد `/readyz` بحالة 200.

## التراجع

عند ظهور خلل حرج:

1. أوقف نشر الإصدار الجديد أو أعد نشر آخر commit مستقر.
2. لا تحذف التصاميم أو النسخ المحلية للمستخدمين.
3. احتفظ بتقرير Playwright وتقرير `production-verification.json` وسجلات `client-error` مع رقم `release`.
4. أصلح المشكلة في Pull Request مستقل وأعد تنفيذ القائمة كاملة.
